What is this document?

An Architecture Decision Record (ADR) captures the reasoning behind each significant technical choice — not just what was decided, but why, what alternatives were considered, and what trade-offs were accepted. This prevents decisions from being re-litigated without understanding the original context, and helps new team members understand the platform quickly.

Each record follows the same structure: Context (the problem) → Decision (what was chosen) → Alternatives (what was considered) → Why (the reasoning) → Trade-offs (what was accepted).

ADR-01
Aurora Serverless v2 + pgvector for data storage and semantic search
Covers: test case storage · run records · vector embeddings · similarity search
✅ Active
Context
The platform needs to store structured test case data (description, steps, tags, run history) and support semantic search — finding similar test cases using natural language queries like "login tests" or "payment flow". These are two very different storage patterns: relational for CRUD, vector for similarity search. Choosing two separate systems adds operational complexity.
Decision
Use Aurora Serverless v2 PostgreSQL with the pgvector extension as the single database for both structured data and vector embeddings. Test cases are stored as rows; their embeddings (generated by Titan Embed v2) are stored as a vector column on the same row.
Alternatives Considered
OptionProsCons
Aurora v2 + pgvector CHOSEN One database for both SQL and vectors. No extra service. pgvector supports cosine similarity natively. Aurora Serverless scales to zero. pgvector is less scalable than dedicated vector DBs at very high volume (millions of vectors).
DynamoDB + Pinecone REJECTED DynamoDB is highly scalable; Pinecone is purpose-built for vectors. Two services to manage. Extra cost. No joins. Pinecone adds another external dependency and monthly cost even at low usage.
DynamoDB + OpenSearch REJECTED Both managed AWS services. OpenSearch supports k-NN vector search. OpenSearch minimum cost is ~$50–100/month even idle. DynamoDB has no native vector support. Complex sync between two stores.
RDS PostgreSQL (provisioned) + pgvector REJECTED Same pgvector capability. Predictable performance. Always-on cost even when not in use (~$30+/month minimum). No auto-pause for a dev/test platform.
Why this choice
  • Single system — test case metadata and its vector embedding live on the same row. No sync, no dual writes, no eventual consistency issues.
  • Scales to zero — Aurora Serverless v2 auto-pauses after 5 minutes of inactivity. For a platform that runs tests periodically (not continuously), this makes idle cost ~$0.
  • pgvector is sufficient — at the scale of test cases (hundreds to low thousands), pgvector with an HNSW index performs sub-second similarity search. A dedicated vector DB would be over-engineering.
  • SQL for everything else — run records, filtering by service/env/tag, aggregations — all naturally fit SQL. DynamoDB would require duplicating data or using scan operations.
Trade-offs Accepted
  • Cold start latency — Aurora takes 8–30 seconds to resume from paused state. First query after idle period is slow. Mitigated with retry logic in Lambda (8 retries × 8s delay).
  • pgvector scale ceiling — if the platform ever holds millions of test vectors, pgvector will be slower than Pinecone/OpenSearch. Acceptable for current scope.
  • No offline/edge access — Aurora is VPC-bound (accessed via Data API). No local dev database without mocking.
ADR-02
RDS Data API — Lambda runs outside the VPC
Covers: Lambda-to-Aurora connectivity · VPC design · cold start performance
✅ Active
Context
Lambda functions need to query Aurora PostgreSQL. The traditional approach is to place Lambda inside the same VPC as Aurora and use a direct TCP connection. However, this has well-known downsides. Aurora Serverless v2 supports an alternative: the RDS Data API, which exposes Aurora over HTTPS without requiring VPC membership.
Decision
Lambda functions (p2t-testcase-writer and p2t-testcase-reader) run outside any VPC and connect to Aurora exclusively through the RDS Data API (rds-data.execute_statement). No VPC configuration on the Lambda functions.
Alternatives Considered
OptionProsCons
RDS Data API (Lambda outside VPC) CHOSEN No VPC config on Lambda. Faster cold starts. Simpler IAM-based auth. No connection pool management. Data API adds ~10–20ms latency per call. Not supported on all Aurora configurations.
Lambda inside VPC + direct TCP REJECTED Lower latency per query. Standard PostgreSQL driver (psycopg2). Connection pooling possible. Lambda cold starts add 1–3 seconds for VPC ENI attachment. Requires NAT Gateway for internet access (~$32/month). Complex subnet/SG config.
Lambda inside VPC + RDS Proxy REJECTED Handles connection pooling. Reduces Aurora connection exhaustion. RDS Proxy costs ~$20+/month. Adds another component. Still requires Lambda in VPC with all its cold start issues.
Why this choice
  • No NAT Gateway cost — Lambda inside a VPC needs a NAT Gateway to reach Bedrock, Secrets Manager, and other AWS services. NAT Gateway costs ~$32/month minimum, plus data transfer. Avoiding it saves significant cost on a dev platform.
  • Faster cold starts — VPC-attached Lambda cold starts are 1–3 seconds longer due to ENI creation. Outside VPC, cold starts are under 500ms.
  • Simpler architecture — no subnet routing, no security group rules between Lambda and Aurora, no connection pool management.
  • Aurora Serverless v2 is designed for this — the Data API is the recommended access pattern for serverless Aurora.
Trade-offs Accepted
  • ~10–20ms extra latency per query vs direct TCP. Acceptable for a test management platform where queries take tens of milliseconds regardless.
  • No persistent connections — each Data API call is stateless. Can't use transactions spanning multiple Lambda invocations.
  • Data API payload limit — 1MB response limit per call. Fine for test case data but would need pagination for very large result sets.
ADR-03
Aurora auto-pause enabled — optimise for cost, not always-on availability
Covers: Aurora Serverless v2 scaling · idle cost · cold start UX impact
⚠️ Trade-off
Context
Aurora Serverless v2 supports auto-pause: if the cluster receives no queries for a configurable period (set to 5 minutes here), it scales down to 0 ACUs and stops billing for compute. The next query wakes it up. This is a deliberate cost-vs-latency trade-off that significantly affects the first user experience after an idle period.
Decision
Aurora auto-pause is enabled with a 5-minute idle threshold. All Lambda functions that query Aurora implement a retry loop (8 retries, 8-second delay) that catches DatabaseResumingException and waits for the cluster to resume before re-attempting the query.
Why this choice
  • Cost — Aurora Serverless v2 at minimum capacity (0.5 ACU) costs ~$0.12/hour = ~$87/month even if unused. Auto-pause brings idle cost to $0. For a dev/test platform used periodically, this is the right trade-off.
  • Transparent to users — the retry logic in Lambda absorbs the resume delay. The UI shows a loading spinner, not an error. Users experience a ~20-second delay on first query after idle, not a failure.
  • Appropriate for usage pattern — test case management is not a 24/7 high-traffic system. Testers use it in bursts during working hours. Long idle periods between sessions make auto-pause ideal.
Trade-offs Accepted
  • First-request latency after idle — 8–30 seconds for Aurora to resume. First page load after an idle period is noticeably slow.
  • Retry complexity in Lambda — every Lambda that queries Aurora must implement the retry pattern. Forgetting this on a new Lambda would cause intermittent 500 errors.
  • Not suitable for production SLA — if this platform is ever used in a production context with SLA requirements, auto-pause should be disabled and minimum ACU set to 0.5 or higher.
ADR-04
Bedrock AgentCore for the AI agent runtime
Covers: agent hosting · tool use · long-running conversations · session management
✅ Active
Context
The AI agent needs to run a Python container (built with the Strands SDK) that: maintains multi-turn conversation context, calls Bedrock Claude for reasoning, uses tools (MCP, SSM lookups, Lambda invocations), and streams responses back to the browser. This is a stateful, long-running workload that doesn't fit neatly into Lambda (15-minute limit, stateless) or a plain HTTP service.
Decision
Use Bedrock AgentCore to host the agent container. AgentCore manages the container lifecycle, session state, and streaming. The agent Docker image is built by CodePipeline and stored in ECR; AgentCore pulls from there.
Alternatives Considered
OptionProsCons
Bedrock AgentCore CHOSEN Built for agentic workloads. Handles session state, streaming, container lifecycle. Integrates natively with Bedrock models and MCP protocol. New service, limited CloudFormation support — must be created via CLI. Less community documentation.
Lambda (Python) REJECTED Simple. Cheap. Auto-scaling. CDK native. 15-minute timeout. Stateless — session context must be stored externally on every call. No native streaming. Cold starts with large Strands/LangChain dependencies.
ECS Fargate (always-on) REJECTED Full container control. No timeout limits. Standard HTTP. Always-on cost (~$30–50/month). Needs load balancer, service discovery. Manual session management. Not purpose-built for AI agents.
Bedrock Agents (managed) REJECTED Fully managed. No container to maintain. Limited to Bedrock-defined action groups. Can't run arbitrary Python code or custom tool integrations. Less flexible for MCP.
Why this choice
  • Purpose-built for agentic workloads — AgentCore handles the hard parts: session persistence, response streaming, container warm-up, and MCP protocol integration.
  • No always-on cost — unlike a standard ECS service, AgentCore only runs when invoked. Billing is per-invocation.
  • Strands SDK alignment — the agent is built with AWS Strands, which is designed to deploy to AgentCore. Using a different runtime would require significant adaptation.
  • Streaming to browser — AgentCore's streaming API lets the UI display the agent's response token by token, giving a ChatGPT-like experience without managing WebSockets manually.
Trade-offs Accepted
  • No CloudFormation support — AgentCore must be created via CLI (one manual step in the deployment). Cannot be fully automated with CDK today.
  • New service risk — AgentCore is a relatively new AWS service. APIs may change, community resources are limited, and behaviour is less predictable than mature services.
  • ECR image dependency — AgentCore must be created after the CodePipeline pushes the first image to ECR. Deployment order matters.
ADR-05
ECS Fargate on-demand for Playwright browser automation
Covers: test execution runtime · browser container · noVNC live view · cost model
✅ Active
Context
Running Playwright requires a full browser (Chromium) in a headless or headed environment — this is a heavy workload (200MB+ memory for a browser process alone). It also needs to serve a live noVNC view so the user can watch the test run in real time in the browser. Lambda cannot run a browser, and an always-on container would waste money between test runs.
Decision
Playwright runs in an ECS Fargate task launched on-demand per test execution (ARM64, 2 vCPU, 4GB RAM). The task starts when a user clicks Run, serves a live browser view via noVNC over WebSocket, executes the test, records the result, then terminates. The UI embeds the noVNC stream in an iframe during execution.
Alternatives Considered
OptionProsCons
ECS Fargate on-demand task CHOSEN Full container. Runs real browser. noVNC live view. Scales to zero between runs. ~$0.10 per test run. 30–60 second cold start to pull image and start container. More complex than Lambda.
Lambda (headless Chrome via layer) REJECTED Faster start. Cheaper per invocation. Simpler deploy. Lambda has 10GB ephemeral storage and 3GB RAM limit — barely enough for Chromium. 15-minute timeout. No live view possible. Unreliable for complex Playwright scripts.
ECS Fargate always-on service REJECTED No cold start. Always ready. Costs ~$30–50/month even when no tests are running. Wasteful for a platform used periodically.
EC2 instance (dedicated) REJECTED Full control. Persistent storage. Always-on cost. Manual maintenance (patching, AMIs). No auto-scaling. Not serverless.
Why this choice
  • Full browser environment — Playwright needs a real Chromium. Only a full container provides this reliably with the memory and CPU headroom needed.
  • Live noVNC view — the test experience is differentiated by showing users what the browser is doing in real time. This requires a persistent WebSocket connection from a running container — impossible with Lambda.
  • Pay per run — a 2-minute test run on 2vCPU/4GB Fargate costs approximately $0.04. Zero cost between runs.
  • Isolation — each test run gets a fresh container, fresh browser state. No cross-contamination between test runs.
Trade-offs Accepted
  • 30–60 second startup time — Fargate task takes time to start. Users see a "starting..." message before the live view appears. Mitigated by showing a progress indicator in the UI.
  • Image pull latency — if the ECR image is large (Playwright + noVNC + Chromium can be 2–3GB), first pull after a long idle period is slower. Fargate caches images at the infrastructure level.
  • No parallel run limit defined — currently no cap on simultaneous Fargate tasks. Heavy concurrent usage could incur unexpected cost.
ADR-06
ARM64 (Graviton) for ECS task definition
Covers: compute architecture · cost · Docker build compatibility
✅ Active
Context
ECS Fargate supports two CPU architectures: x86_64 (Intel/AMD) and ARM64 (AWS Graviton). The choice affects cost, performance, and — critically — Docker image compatibility. The Docker image must be built for the same architecture as the task definition.
Decision
ECS task definition uses ARM64 (Graviton2). The CodeBuild pipeline builds the Docker image with --platform linux/arm64. Fargate Graviton2 pricing is ~20% cheaper than x86_64 for the same vCPU/memory.
Why this choice
  • ~20% cost reduction — Graviton2 Fargate pricing is lower per vCPU-hour and per GB-hour than x86_64. For a 2vCPU/4GB task, this is a meaningful saving per test run.
  • Playwright supports ARM64 — Playwright and Chromium have stable ARM64 Linux builds. No compatibility issues.
  • Performance parity — for I/O-bound workloads like browser automation, ARM64 and x86_64 perform equivalently.
Trade-offs Accepted
  • Build pipeline must target ARM64 — CodeBuild must use a build environment with ARM64 support (LINUX_ARM_CONTAINER), or use Docker buildx with cross-compilation. If this is misconfigured, the container will fail to start with an "exec format error".
  • Some native packages may lack ARM64 builds — if future dependencies don't have ARM64 packages, the architecture choice would need to be revisited.
ADR-07
4 separate GitHub repos — not a monorepo
Covers: repo structure · deploy cadence · pipeline triggers · team ownership
✅ Active
Context
The platform has four distinct components with very different technology stacks, build tools, deploy targets, and change cadences: the React UI, the Python Lambda functions, the AI agent container, and the Playwright automation container. Deciding whether to keep these in one repository or separate ones has significant implications for CI/CD, team workflow, and deployment speed.
Decision
Four separate GitHub repositories: Prompt2TestUI (React + CDK), Prompt2TestLambda (Python), Prompt2TestAgent (agent container), Prompt2TestPlaywrightMCP (Playwright container). Each repo has its own pipeline that deploys independently on push.
Alternatives Considered
OptionProsCons
4 separate repos CHOSEN Independent deploys. Each repo has its own history. Changing Playwright doesn't trigger UI rebuild. Smaller clone size. Clear ownership boundaries. Cross-cutting changes require PRs in multiple repos. Harder to do atomic changes across components.
Monorepo (all in one) REJECTED One place for all code. Atomic cross-component changes. Shared tooling. A typo fix in the UI would trigger a full Lambda + Docker rebuild. Pipeline logic becomes complex (path filters). Slower CI for unrelated changes.
Why this choice
  • Independent deploy cadences — the UI changes frequently (daily). Lambda logic changes occasionally. The Playwright container rarely changes. Separate repos means each deploys at its own pace without affecting others.
  • Clear boundaries — each repo corresponds to one deploy target (Amplify, Lambda, ECR×2). This makes it obvious what a push to each repo does.
  • Smaller blast radius — a broken push to Prompt2TestAgent doesn't affect the UI or Lambda. Failures are isolated.
  • Different tech stacks — TypeScript/React, Python, and Dockerfile-based containers have no shared build tooling. A monorepo would give the appearance of unity without the benefits.
Trade-offs Accepted
  • Cross-repo changes need coordination — if a new Lambda API requires a UI change, two PRs in two repos must be coordinated and deployed in the right order.
  • 4 repos to keep up to date — each repo needs its own GitHub Actions / CodePipeline setup, README, and branch protection rules.
ADR-08
AWS Amplify for React UI hosting
Covers: static hosting · CDN · CI/CD for frontend · env var management
✅ Active
Context
The React UI is a static single-page application (SPA) built with Vite. It needs to be hosted on a CDN, rebuilt automatically when code is pushed to GitHub, and have environment variables injected at build time (Cognito IDs, region, AgentCore ARN).
Decision
Use AWS Amplify Hosting. CDK creates the Amplify app and sets all environment variables. GitHub branch connection (OAuth) is done manually once. After that, every push to master triggers an automatic build and deploy.
Alternatives Considered
OptionProsCons
AWS Amplify Hosting CHOSEN Managed CI/CD for frontend. Built-in CDN (CloudFront). Branch previews. Env var management. CDK-provisionable. Free tier covers typical usage. GitHub OAuth connection cannot be automated (requires browser). Amplify build environment is opinionated.
S3 + CloudFront (manual) REJECTED Full control. CDK native. No OAuth step. Must build a CI/CD pipeline manually (CodePipeline or GitHub Actions) to upload to S3 on push. More infrastructure to maintain. No automatic branch previews.
Vercel / Netlify REJECTED Excellent DX. Fast deploys. Free tier. External service outside AWS. Env vars with AWS credentials/IDs stored outside AWS. Adds external vendor dependency.
Why this choice
  • Zero CI/CD work for the frontend — Amplify handles build, deploy, and CDN invalidation automatically on every push. No CodePipeline to configure for the UI.
  • CDK can provision it — the Amplify app, environment variables (Cognito IDs, region), and build settings are all set by CDK. Only the GitHub OAuth step is manual.
  • Free tier — Amplify free tier (1,000 build minutes/month, 15GB serving) covers typical dev usage at zero cost.
  • Stays within AWS — all config stays inside the AWS account. No env vars leave the environment.
Trade-offs Accepted
  • OAuth step cannot be automated — connecting Amplify to GitHub requires a browser-based OAuth flow. This is the one step in the deployment guide that cannot be scripted.
  • Amplify build is opinionated — Amplify expects a specific build output structure. The amplify.yml must match the Vite output directory (web/dist).
ADR-09
Amazon Cognito for authentication
Covers: user login · session tokens · browser-to-AWS SDK calls · IAM federation
✅ Active
Context
The React UI needs to: authenticate users, call AWS services (Lambda, AgentCore, ECS) directly from the browser using the AWS SDK, and do so securely without embedding AWS credentials in frontend code. This requires both identity (who is this user?) and authorisation (what AWS resources can they access?).
Decision
Use Cognito User Pool for login (username/password, session management) and Cognito Identity Pool to exchange the User Pool JWT for temporary AWS credentials. The browser SDK uses these credentials to call Lambda, AgentCore, and ECS directly — no backend proxy needed.
Why this choice
  • Browser-to-AWS without a backend proxy — Cognito Identity Pool issues temporary STS credentials to the browser. The React app can call Lambda, ECS, and AgentCore directly using aws-sdk without routing everything through an API Gateway or backend server.
  • No credentials in frontend code — credentials are short-lived (1 hour) and scoped to the Cognito Auth Role. No static AWS keys anywhere in the UI.
  • Fully managed — Cognito handles password hashing, session tokens, MFA (if needed later), and JWT validation. No custom auth code to maintain.
  • CDK native — User Pool, Identity Pool, and Auth Role are all provisionable with CDK in a few lines.
Trade-offs Accepted
  • Cognito UX is basic — the default Cognito hosted UI and flows are functional but not polished. Custom login UI requires more work.
  • FORCE_CHANGE_PASSWORD on new users — admin-created users must change password on first login. This is expected Cognito behaviour but must be handled in the UI flow.
  • Identity Pool complexity — the User Pool + Identity Pool + Auth Role chain can be confusing to debug when permissions issues arise.
ADR-10
Lambda deployed via separate CodePipeline, not bundled in CDK
Covers: Lambda code lifecycle · CDK asset management · repo separation
✅ Active
Context
CDK can deploy Lambda code directly using Code.fromAsset() — bundling the Python files at deploy time. But Lambda code lives in a separate GitHub repo (Prompt2TestLambda) with its own change cadence. Developers should be able to push Lambda code changes without running cdk deploy.
Decision
CDK creates Lambda functions with placeholder code (a stub that returns a 503). A separate CodePipeline (prompt2test-lambda) watches Prompt2TestLambda on GitHub and runs aws lambda update-function-code on every push to deploy real code.
Why this choice
  • Lambda code has its own cadence — business logic in Lambda changes frequently and independently of infrastructure. Developers pushing a bug fix to Prompt2TestLambda should not need to run cdk deploy.
  • CDK deploy is slow — cdk deploy takes 5–10 minutes. update-function-code takes under 30 seconds. Using the pipeline for Lambda updates is dramatically faster for day-to-day development.
  • Separation of concerns — infrastructure team manages CDK; backend team manages Lambda code. These teams (or concerns) don't need to coordinate on every deploy.
Trade-offs Accepted
  • Two-step initial deployment — after cdk deploy, Lambda functions return 503 until the CodePipeline runs for the first time. Must remember to trigger the pipeline after initial CDK deploy.
  • Lambda code not in CDK state — CloudFormation does not track the Lambda code version. If the stack is torn down and redeployed, the pipeline must run again to restore the real code.
ADR-11
No API Gateway — Lambda invoked directly from the browser
Covers: API design · browser-to-Lambda call pattern · latency · cost
✅ Active
Context
The React UI needs to call p2t-testcase-writer and p2t-testcase-reader to save and fetch test cases. The typical pattern is to put API Gateway in front of Lambda to expose it as an HTTP endpoint. However, the React app already has AWS credentials from Cognito Identity Pool — meaning it can call Lambda directly using the AWS SDK.
Decision
The React UI calls Lambda using lambda.invoke() from the AWS SDK with the Cognito-issued credentials. No API Gateway. No HTTP REST or GraphQL endpoint. Lambda is invoked directly as an internal AWS API call.
Alternatives Considered
OptionProsCons
Direct Lambda invoke (SDK) CHOSEN No extra service. No cost per request beyond Lambda invocation. Already have AWS credentials from Cognito. No CORS config needed. Tightly coupled to AWS SDK. Cannot be called from non-AWS clients without credentials. Not a standard REST API.
API Gateway REST API REJECTED Standard HTTP. Callable from any client. Built-in throttling, caching, request validation. $3.50 per million requests. Extra latency hop. Requires CORS configuration. Adds CDK complexity. Unnecessary since the only client is the internal React app with AWS credentials.
API Gateway HTTP API REJECTED Cheaper ($1/million). Lower latency than REST API. Still an extra service and hop. Same reasoning as REST API — unnecessary when the client already has AWS credentials.
Why this choice
  • The client already has AWS credentials — Cognito Identity Pool gives the browser temporary STS credentials. Using them to call Lambda directly is the natural pattern. API Gateway would be an unnecessary layer.
  • Cost — direct Lambda invoke is charged only for Lambda execution time. API Gateway adds per-request charges on top. For a low-traffic internal tool, this matters.
  • Latency — removing API Gateway removes one network hop and one AWS service from the call chain.
  • No CORS — API Gateway requires CORS headers configuration. Direct Lambda invoke through the SDK bypasses CORS entirely.
Trade-offs Accepted
  • Not a standard REST API — Lambda cannot be called from Postman, curl, or other tools without AWS credentials. Fine for an internal tool; would be a blocker if external integrations were needed.
  • AWS SDK dependency in UI — the React app imports and uses aws-sdk. This increases bundle size and couples the frontend to AWS.
  • No built-in throttling — API Gateway provides per-client throttling out of the box. Without it, a misbehaving UI could flood Lambda. Lambda's own concurrency limits are the only protection.
ADR-12
SSM Parameter Store for runtime configuration
Covers: service URLs · Playwright config · environment-specific settings · ECS task config
✅ Active
Context
The platform needs to store runtime configuration that varies per environment (dev/staging/prod): service names and URLs, Playwright cluster details, base URLs, timeouts, and OAuth endpoints. This config must be readable by the Lambda functions, the ECS Fargate tasks, and the React UI — and must be updatable without redeploying code.
Decision
All runtime configuration is stored in SSM Parameter Store under a structured path hierarchy: /prompt2test/config/{env}/services/{ServiceName}/URL. ECS task definitions reference SSM paths directly. Lambda and the UI read parameters at runtime via the SSM API.
Alternatives Considered
OptionProsCons
SSM Parameter Store CHOSEN Hierarchical path structure. IAM-controlled access. Free for standard parameters. ECS can inject SSM params as env vars natively. CDK-provisionable. Extra API call to read params at runtime. No secret rotation (use Secrets Manager for that).
Lambda/ECS environment variables REJECTED Zero latency — no API call needed. Changing a config value requires redeploying the Lambda or ECS task definition. Not suitable for values that change independently of code (e.g. service URLs).
Secrets Manager REJECTED Built-in rotation. Encrypted at rest. Good for credentials. $0.40/secret/month. Designed for secrets (passwords, API keys), not general config. Overkill for non-sensitive values like service URLs and timeouts.
DynamoDB config table REJECTED Flexible schema. Queryable. Another service to provision and maintain. No IAM path-based access control. SSM already does this job well.
Why this choice
  • Update config without code deploy — adding a new service URL or changing a timeout is an aws ssm put-parameter command. No Lambda redeploy, no CDK run, no pipeline trigger.
  • Hierarchical paths — the /prompt2test/config/dev/services/*/URL pattern lets the UI discover all services for an environment with a single get-parameters-by-path call.
  • ECS native integration — ECS task definitions can reference SSM parameters directly as environment variables, injected securely at task launch time.
  • Free for standard tier — non-secret configuration parameters are free in SSM. Secrets Manager would cost $0.40/parameter/month.
Trade-offs Accepted
  • Runtime API call overhead — Lambda reads SSM on each invocation (or caches with a TTL). Adds ~5–10ms per parameter read. Acceptable for this use case.
  • Not for secrets — SSM standard parameters are not encrypted by default. Sensitive values (passwords, tokens) use Secrets Manager instead (prompt2test/aurora/credentials). The distinction must be maintained as the platform grows.
