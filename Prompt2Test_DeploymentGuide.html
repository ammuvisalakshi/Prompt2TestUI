Prompt2Test — New AWS Account Deployment Guide

Complete step-by-step walkthrough with expected output after every step · Brand new AWS account → fully running platform
⏱ Estimated Time: 45–60 minutes ☁ AWS Region: us-east-1 📅 Updated: 2026-03-25
Understanding the two parts of this deployment

infrastructure/ folder = CDK code (the automated installer) — TypeScript code that, when you run one command (cdk deploy), automatically creates all 16 AWS resources (Cognito, Aurora, Lambda, ECS, ECR, SSM, CodePipeline, Amplify, IAM roles) in your account. You do not click anything in the console for these.

Manual steps (3 total) — Three things CDK cannot automate: enabling Bedrock model access, creating the AgentCore runtime (no CloudFormation support yet), and connecting Amplify to GitHub (requires OAuth). This guide walks through both in the correct order.

⚠️

If you are the root user of a brand new account, you already have full access. If you created a separate IAM user, attach AdministratorAccess to it first: AWS Console → IAM → Users → your user → Add permissions → AdministratorAccess. Without it, cdk deploy will fail with "Access Denied".

Phase 1
Local Machine Setup
Install the tools you need on your computer — do this before touching AWS
1
Install AWS CLI
The command-line tool for talking to AWS from your terminal

Download and install from aws.amazon.com/cli — choose your operating system (Windows / Mac / Linux). After install, open a new terminal and run:

aws --version
✅
What you should see
aws-cli/2.15.0 Python/3.11.6 Windows/10 exe/AMD64 prompt/off

Version number 2.x.x confirms it is installed. If you see command not found the install did not complete — try opening a new terminal window first.

2
Install Node.js (version 18 or higher)
Required to run CDK — download from nodejs.org → LTS version
node --version
✅
What you should see
v20.11.0

Any version v18.x.x or higher is fine. If it shows v16 or lower, download the latest LTS from nodejs.org and reinstall.

3
Install AWS CDK globally
The tool that reads your infrastructure code and deploys it to AWS
npm install -g aws-cdk
cdk --version
✅
What you should see
2.170.0 (build abc1234)

Any 2.x.x version is good. The install itself prints many lines of progress — that is normal. Only the final cdk --version output matters.

8
Bootstrap CDK in the new account
One-time setup — creates an S3 bucket and IAM roles that CDK needs internally

First, get your account ID:

aws sts get-caller-identity --query Account --output text

Then run bootstrap — replace 123456789012 with your actual account ID:

cdk bootstrap aws://123456789012/us-east-1

This takes about 1–2 minutes.

✅

The green bootstrapped message confirms success. You will also see a new CloudFormation stack named CDKToolkit appear in your account — this is normal.

If you see "Access Denied"

Your IAM user does not have enough permissions. Go to AWS Console → IAM → Users → your user → Add permissions and attach the AdministratorAccess managed policy.

Phase 3
CDK Deploy — Creates All 17 AWS Resources Automatically
Two commands. Takes about 5–10 minutes. Creates everything in the "What Gets Deployed" table above.
10
Configure cdk.json with your GitHub details
Tell CDK your GitHub username and the connection ARN from Step 9

Open the file Prompt2TestUI/infrastructure/cdk.json in any text editor and replace the two placeholder values:

{
  "app": "npx ts-node --prefer-ts-exts bin/infrastructure.ts",
  "context": {
    "githubOwner": "your-actual-github-username",
    "githubConnectionArn": "arn:aws:codeconnections:us-east-1:123456789012:connection/xxxxxxxx"
  }
}
  • githubOwner — your GitHub username (e.g. jsmith, not the full URL)
  • githubConnectionArn — the ARN you copied in Step 9. It looks like arn:aws:codeconnections:us-east-1:<accountId>:connection/<uuid>

Save the file after editing.

✅
What to verify

The file should have your real GitHub username (e.g., ammuvisalakshi) and the full connection ARN — no placeholder text like YOUR_ remaining.

11
Run CDK Deploy
This is the main install step — creates all 15 AWS resources automatically

cd infrastructure
npm install

This deployment will make potentially sensitive changes according to your current security approval level. Please confirm you intend to make the following modifications: IAM Statement Changes ... Do you wish to deploy these changes (y/n)? y

Type y and press Enter. Then wait — it runs for about 5–10 minutes.

CDK automatically rolls back on failure. Read the error message — the most common causes are: wrong account permissions, or the Aurora version not available in your region. Fix the issue and run cdk deploy again — CDK picks up from where it can.

Phase 5
Create Bedrock AgentCore Runtime — Manual CLI Step
AWS does not support AgentCore in CloudFormation/CDK yet — this one step must be done via CLI
13
Create the AgentCore Runtime
Uses the ECR agent image (Phase 4) and IAM role (CDK output) to spin up the AI agent container

Use the two values from your CDK outputs (Step 11). Replace both placeholders:

aws bedrock-agentcore-control create-agent-runtime \
  --agent-runtime-name Prompt2TestAgent \
  --agent-runtime-artifact \
    '{"containerConfiguration":{"image":"YOUR_OutAgentEcrUri:latest"}}' \
  --role-arn YOUR_OutAgentCoreRoleArn \
  --network-configuration '{"networkMode":"PUBLIC"}'

Example with real values filled in:

aws bedrock-agentcore-control create-agent-runtime \
  --agent-runtime-name Prompt2TestAgent \
  --agent-runtime-artifact \
    '{"containerConfiguration":{"image":"123456789012.dkr.ecr.us-east-1.amazonaws.com/prompt2test-agent:latest"}}' \
  --role-arn arn:aws:iam::123456789012:role/prompt2test-agentcore-role \
  --network-configuration '{"networkMode":"PUBLIC"}'
✅
What you should see

The command immediately returns a JSON response:

{ "agentRuntimeArn": "arn:aws:bedrock:us-east-1:123456789012:agent-runtime/XXXXXXXXXX", "agentRuntimeId": "XXXXXXXXXX", "agentRuntimeName":"Prompt2TestAgent", "status": "CREATING" }
📋

Copy the agentRuntimeArn value. You need it in the next step to configure the UI.

The status CREATING is normal — it will become ACTIVE within a few minutes. Poll for ACTIVE status with the agentRuntimeId from above:

aws bedrock-agentcore-control get-agent-runtime \
  --agent-runtime-id XXXXXXXXXX \
  --query 'status' --output text

Repeat every 30 seconds until the output is ACTIVE. Then proceed to the next step.

If you get "command not found: bedrock-agentcore-control"

Your AWS CLI version is too old. Run aws --version — you need version 2.15 or higher. Update your AWS CLI from aws.amazon.com/cli.

Phase 7
Add Your Services and Create Admin User
Add your application services to SSM so they appear in the UI, then create your first login
15
Create teams and configure services via the UI
No AWS CLI commands needed — everything is managed through the Prompt2Test UI

After logging in as admin (Step 16), use the UI to set up your teams and service configs. Service parameters save directly to DynamoDB — no AWS Console or CLI required.

1. Members page → Create teams and invite users

  1. Go to the Members page (admin only)
  2. Click Create Team → enter a team name (e.g. teamA) → confirm
  3. Click Invite User → enter the email address → the user is created in Cognito and added to the team group
  4. Repeat for additional teams and members

2. Config & Accounts page → Add service parameters

  1. Log in as a team member (or switch to a team account)
  2. Go to the Config & Accounts page
  3. Select an environment (e.g. dev) from the environment picker
  4. Click Add Service → enter a service name (e.g. MyService)
  5. Add parameters for the service: URL, API_KEY, or any key-value pair
  6. Parameters save to DynamoDB with key pk=SERVICE#{team}#{env}, sk={svc}#{key}
  7. Repeat for each environment (staging, prod, etc.)
✅
What you should see

After adding services, they appear as selectable chips in the Agent page when authoring test cases. You can verify the DynamoDB items were created in AWS Console → DynamoDB → Tables → prompt2test-config → Explore items.

16
Create your first admin user and the admin Cognito Group
Use the User Pool ID from your CDK outputs (Step 11) · The admin group is required for login

First, create the admin Cognito Group. Users not in any group cannot log in — the admin group grants access to the Members page for team management.

# Step 1: Create the admin Cognito Group
aws cognito-idp create-group \
  --user-pool-id YOUR_OutUserPoolId \
  --group-name admin \
  --region us-east-1

# Step 2: Create the admin user
aws cognito-idp admin-create-user \
  --user-pool-id YOUR_OutUserPoolId \
  --username admin@yourcompany.com \
  --temporary-password TempPass123! \
  --user-attributes \
    Name=email,Value=admin@yourcompany.com \
    Name=email_verified,Value=true

# Step 3: Add the admin user to the admin group
aws cognito-idp admin-add-user-to-group \
  --user-pool-id YOUR_OutUserPoolId \
  --username admin@yourcompany.com \
  --group-name admin \
  --region us-east-1

The admin group is required — users not in any group cannot log in. Admin users see only the Members page for team management. To add team members later, use the Members page in the UI (Step 15) — no CLI needed.

✅
What you should see
{ "User": { "Username": "admin@yourcompany.com", "UserStatus": "FORCE_CHANGE_PASSWORD", "Enabled": true } }

The status FORCE_CHANGE_PASSWORD is correct and expected — the user will be asked to set a new permanent password the first time they log in. The create-group and admin-add-user-to-group commands return empty output on success.

Phase 8
Verify the Full Platform End-to-End
Run through each screen to confirm everything is working correctly
17
Log in and test the full flow
  1. Open your Amplify URL (from Step 14) in a browser
  2. Log in with admin@yourcompany.com and the temporary password TempPass123!
  3. Set a new password when prompted — choose something permanent
  4. Go to Config & Accounts tab — add a service via the UI (Step 15) — services appear as chips in the Agent page
  5. Go to Agent tab → select environment and a service → type a test prompt like "test that the login page loads" → the agent should respond and generate a test plan
  6. Save the test case → click Save → fill in the form → click Confirm → click View Test Case → to open the test case page
  7. Click Automate on the test case page → a live browser window should open showing the agent automating the test → after it finishes, review the captured steps
  8. Click Save Steps to persist the captured Playwright MCP calls for deterministic replay
  9. Click Run Test → the test replays deterministically (no LLM) → after it finishes, a PASS or FAIL result should be recorded
  10. Go to Inventory tab → your saved test case should appear in the list with the run result
✅
What success looks like for each check
✅ Login page loads → Amplify hosting is working ✅ Password change prompted → Cognito User Pool is working ✅ Services appear in Config → DynamoDB prompt2test-config is working ✅ Agent responds to prompt → AgentCore runtime is active, IAM roles correct ✅ Test case saves to Inventory → Lambda + Aurora is working ✅ Automate opens live browser → ECS Fargate + noVNC is working ✅ Steps captured and saved → playwright_calls persisted to Aurora ✅ PASS/FAIL result recorded → Full end-to-end flow complete (LLM automate + deterministic replay)
Agent does not respond — most common cause

The AgentCore runtime may still be in CREATING state. Wait 2–3 more minutes and try again. You can check its status in AWS Console → Bedrock → AgentCore runtimes. It should show ACTIVE.
