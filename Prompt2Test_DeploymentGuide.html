Prompt2Test — New AWS Account Deployment Guide

Complete step-by-step guide to deploy the entire platform from scratch on any AWS account
Estimated Time: 45–60 minutes AWS Region: us-east-1 Updated: 2026-03-25
What does the CDK Infrastructure do?

This project has a folder called infrastructure/ which contains CDK (Cloud Development Kit) code — TypeScript code that, when you run a single command (cdk deploy), automatically creates all 15 AWS resources in your account.

Think of it as an installer. You don't have to manually create each service in the console — CDK does it all for you. However, there are 3 manual steps that CDK cannot automate: enabling Bedrock model access, creating the AgentCore runtime, and connecting Amplify to GitHub.

This guide walks you through everything — the automated parts and the manual parts — in the right order.

Phase 1
Local Machine Setup
Install the tools you need on your computer before touching AWS
1
Install AWS CLI
Lets you run AWS commands from your terminal

Download and install from the AWS website, then verify:

# Verify install
aws --version
# Should output: aws-cli/2.x.x ...

If not installed: download from aws.amazon.com/cli — choose Windows, Mac, or Linux installer.

2
Install Node.js (v18 or higher)
Required for CDK
node --version
# Should output: v18.x.x or higher

If not installed: download from nodejs.org → LTS version.

3
Install AWS CDK globally
The tool that deploys your infrastructure
npm install -g aws-cdk
cdk --version
# Should output: 2.x.x
4
Configure AWS credentials for the new account
Tells AWS CLI which account to talk to

First, create an access key in the new AWS account:

Log into the new AWS account in a browser

Go to IAM → Users → your user → Security credentials

Click Create access key → select CLI → copy both values

Then configure the CLI:

aws configure

# Enter when prompted:
AWS Access Key ID     → AKIAIOSFODNN7EXAMPLE
AWS Secret Access Key → wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
Default region name   → us-east-1
Default output format → json

Verify it works:

aws sts get-caller-identity
# Should show your new account's Account ID, UserId, Arn
5
Clone the repository
Get the project code onto your machine
git clone https://github.com/YOUR_GITHUB_USERNAME/Prompt2TestUI
cd Prompt2TestUI
Phase 2
Prepare the AWS Account (Manual — Console Steps)
Three things that must be done in the AWS Console before running CDK
6
Enable Bedrock Model Access
Must be done first — can take a few minutes to activate

Do this first — Bedrock model access takes a few minutes to become active. Start it now so it is ready by the time you need it.

Go to AWS Console → Amazon Bedrock

In the left sidebar click Model access

Click Manage model access (top right)

Check these two models:
  • Claude Sonnet 4.5 (under Anthropic — use cross-region inference profile)
  • Titan Embed Text v2 (under Amazon)

Click Save changes

Wait until both show Access granted (usually 1–5 minutes)

7
Bootstrap CDK in the new account
One-time setup — creates internal CDK resources (S3 bucket, IAM roles)

First get your account ID:

aws sts get-caller-identity --query Account --output text
# Outputs something like: 123456789012

Then bootstrap:

cdk bootstrap aws://123456789012/us-east-1
# Replace 123456789012 with your actual account ID

You should see: Environment aws://123456789012/us-east-1 bootstrapped.

8
Create a GitHub CodeStar Connection
Lets AWS CodePipeline pull code from your GitHub repos automatically

In the AWS Console, search for "Developer Tools" in the top search bar

Go to Settings → Connections

Click Create connection

Select GitHub → click Connect to GitHub

Authorize the OAuth flow — GitHub will ask you to approve AWS access

Click Connect — the status will show Available

Click on the connection and copy the Connection ARN

📋

Save this ARN — you need it in the next step.
It looks like: arn:aws:codestar-connections:us-east-1:123456789012:connection/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

Phase 3
CDK Deploy — Automated (Creates All 15 Resources)
One command that provisions everything. Takes about 5–10 minutes.
9
Configure cdk.json with your GitHub details
Tell CDK where your GitHub repos are

Open the file infrastructure/cdk.json and update the two values:

{
  "app": "npx ts-node --prefer-ts-exts bin/infrastructure.ts",
  "context": {
    "githubOwner": "your-actual-github-username",
    "githubConnectionArn": "arn:aws:codestar-connections:us-east-1:123456789012:connection/xxxxxxxx"
  }
}

Replace your-actual-github-username with your GitHub username, and paste in the Connection ARN from Step 8.

10
Run CDK Deploy
This creates all 15 AWS resources automatically
cd infrastructure
npm install
cdk deploy

CDK will show you a list of IAM changes it will make and ask for confirmation:

Do you wish to deploy these changes (y/n)? y

Then it runs for about 5–10 minutes. You will see progress like:

Prompt2TestStack: deploying... ✅ Prompt2TestStack Outputs: Prompt2TestStack.OutUserPoolId = us-east-1_XXXXXXXXX Prompt2TestStack.OutUserPoolClientId = 1bvraf... Prompt2TestStack.OutIdentityPoolId = us-east-1:xxxx-xxxx-... Prompt2TestStack.OutRegion = us-east-1 Prompt2TestStack.OutAgentCoreRoleArn = arn:aws:iam::123456789012:role/... Prompt2TestStack.OutAgentEcrUri = 123456789012.dkr.ecr.us-east-1.amazonaws.com/prompt2test-agent Prompt2TestStack.OutAmplifyAppId = d1234abcd
📋

Copy and save all of these output values. You will need them in the next steps. You can also retrieve them later from: AWS Console → CloudFormation → Prompt2TestStack → Outputs

Phase 4
Build & Push Docker Images
Trigger the CodePipelines to build and push the agent and Playwright containers to ECR
11
Trigger both CodePipelines
CDK created the pipelines but they haven't run yet
aws codepipeline start-pipeline-execution --name prompt2test-playwright-mcp
aws codepipeline start-pipeline-execution --name prompt2test-agent

Then monitor progress:

Go to AWS Console → CodePipeline

You should see both pipelines running

Wait until both show Succeeded — each takes about 5–10 minutes

If a pipeline fails, click on the failed stage → View in CodeBuild to see the build logs. The most common issue is a missing Dockerfile in the GitHub repo.

Phase 5
Create Bedrock AgentCore Runtime (Manual — CLI)
AWS does not support AgentCore in CloudFormation/CDK yet — this must be done via CLI
12
Create the AgentCore Runtime
Uses the ECR image (from Step 11) and the IAM role (from CDK output)

Use the values from Step 10 outputs — replace both placeholders:

aws bedrock-agentcore-control create-agent-runtime \
  --agent-runtime-name Prompt2TestAgent \
  --agent-runtime-artifact \
    '{"containerConfiguration":{"image":"YOUR_OutAgentEcrUri:latest"}}' \
  --role-arn YOUR_OutAgentCoreRoleArn \
  --network-configuration '{"networkMode":"PUBLIC"}'

Example with real values:

aws bedrock-agentcore-control create-agent-runtime \
  --agent-runtime-name Prompt2TestAgent \
  --agent-runtime-artifact \
    '{"containerConfiguration":{"image":"123456789012.dkr.ecr.us-east-1.amazonaws.com/prompt2test-agent:latest"}}' \
  --role-arn arn:aws:iam::123456789012:role/prompt2test-agentcore-role \
  --network-configuration '{"networkMode":"PUBLIC"}'

The response JSON will include the runtime ARN:

{ "agentRuntimeArn": "arn:aws:bedrock:us-east-1:123456789012:agent-runtime/XXXXXXXXXX", "agentRuntimeId": "XXXXXXXXXX", "status": "CREATING" }
📋

Copy the agentRuntimeArn — you need it in the next step to configure Amplify.

Phase 6
Connect Amplify to GitHub (Manual — Console)
CDK created the Amplify app but GitHub OAuth requires a manual authorization step
13
Connect Amplify to your GitHub repo and set the AgentCore ARN
Final wiring step — after this your UI is live

Go to AWS Console → AWS Amplify

Click on Prompt2TestUI

Click Connect branch

Select GitHub → authorize if prompted → select repo Prompt2TestUI → branch master

Under Environment variables, find VITE_AGENT_RUNTIME_ARN and update it with the ARN from Step 12

Click Save and deploy

Wait 3–5 minutes for the build to complete

When done, your app URL appears at the top of the Amplify console:
https://master.xxxxxxxxx.amplifyapp.com

All other environment variables (Cognito IDs, region, identity pool) were already set by CDK when it created the Amplify app. You only need to set VITE_AGENT_RUNTIME_ARN.

Phase 7
Add Your Services & Create Users
Configure your environment-specific data so the platform is usable
14
Add Services to SSM Parameter Store
These appear as chips in the UI when selecting a service to test

The parameter naming pattern is: /prompt2test/config/{env}/services/{ServiceName}/URL

# Add services for dev environment
aws ssm put-parameter \
  --name "/prompt2test/config/dev/services/MyService/URL" \
  --value "https://myservice.dev.example.com" \
  --type String

# Add another service
aws ssm put-parameter \
  --name "/prompt2test/config/dev/services/PaymentsAPI/URL" \
  --value "https://payments.dev.example.com" \
  --type String

# Add for staging environment
aws ssm put-parameter \
  --name "/prompt2test/config/staging/services/MyService/URL" \
  --value "https://myservice.staging.example.com" \
  --type String

Repeat for each service and each environment (dev, staging, prod, etc.) you want to test.

15
Create the First Admin User
Use the User Pool ID from the CDK outputs (Step 10)
aws cognito-idp admin-create-user \
  --user-pool-id YOUR_OutUserPoolId \
  --username admin@yourcompany.com \
  --temporary-password TempPass123! \
  --user-attributes \
    Name=email,Value=admin@yourcompany.com \
    Name=email_verified,Value=true

The user will be prompted to change their password on first login.

To add more users (team members), repeat this command with their email address.

Phase 8
Verify Everything Works End-to-End
Run through these checks to confirm the deployment is successful
16
End-to-End Verification Checklist
  1. Open your Amplify URL in a browser
  2. Log in with admin@yourcompany.com and the temporary password
  3. Set a new permanent password when prompted
  4. Go to Config → confirm your services appear as chips
  5. Go to Agent → select environment and service → type a test prompt → confirm the agent responds and generates a plan
  6. Save the test case → go to Inventory → confirm it appears in the list
  7. Click Run on a test case → confirm the browser automation starts and the live view opens
  8. After execution, confirm PASS/FAIL result is recorded in the run history
