Technical Guide

How Prompt2Test Works

A complete walkthrough of the platform architecture, the Plan Mode workflow, and the Automate Mode workflow — from user prompt to live browser execution.

1

Overall Platform Architecture

Prompt2Test is a fully serverless AI-powered test automation platform built on AWS. Users describe a test in plain English. The AI agent plans and executes it in a real browser, while the user watches live. No test code needs to be written.

🖥️

Frontend

React app on AWS Amplify. Cognito handles login. Users interact via a chat-style interface.

🤖

AI Agent

Python Strands agent on AgentCore. Uses Claude Sonnet 4.5 on Bedrock for all AI reasoning.

🌐

Browser Automation

playwright-mcp on ECS Fargate. Runs real Chromium headed browser. Viewed live via noVNC.

🔄

CI/CD

CodePipeline + CodeBuild auto-deploy Docker images to ECR and ECS on every GitHub push.

2

Plan Mode Workflow

When the user types a test description and clicks Plan, the system generates a structured, step-by-step test plan using Claude. No browser is opened — this is pure AI reasoning. The result is a human-readable plan the user can review and refine before execution.

Step Service What happens Browser? MCP call?
1 User types prompt Amplify Frontend User describes the test in plain English and clicks Plan ❌❌
2 HTTP POST AgentCore Frontend sends { mode:"plan", inputText:"..." } signed with Cognito SigV4 credentials ❌❌
3 agent_runner.plan() AgentCore Python Python agent formats the prompt and sends it to Claude via Strands SDK ❌❌
4 Claude reasons Amazon Bedrock Claude Sonnet 4.5 reads the prompt and generates a structured JSON plan with steps, tools, and estimates ❌❌
5 Plan displayed Amplify Frontend React renders the plan as readable steps. User reviews and types "run" to execute ❌❌
3

Automate Mode Workflow

When the user types run, the system executes the plan in a real Chromium browser. Claude acts as an autonomous agent — it thinks, calls a browser tool, observes the result, and repeats until all steps are done. The user watches the browser execute in real time via the noVNC live panel.

Step Service What happens Browser? MCP call?
1 User says "run" Amplify Frontend noVNC panel appears. Frontend sends { mode:"automate", plan:{...} } to AgentCore ✅ opens❌
2 MCP client connects AgentCore → playwright-mcp Python agent opens SSE connection to playwright-mcp via ALB:3000. Host header set to localhost:3000 to bypass CSRF check ✅✅ connect
3 Agent loop begins Bedrock Claude + Strands Claude reads the plan and decides which MCP tool to call for step 1 ✅✅
4 Browser action playwright-mcp → Chromium playwright-mcp executes the tool (navigate, click, fill, assert, screenshot) in the real Chromium browser on Xvfb ✅ action✅ per step
5 Live view streams x11vnc → websockify → CloudFront → noVNC x11vnc captures the Xvfb screen, websockify converts to WebSocket, CloudFront delivers to the noVNC iframe. User watches live. ✅ visible❌
6 Result → Claude Bedrock Claude playwright-mcp returns the tool result. Claude decides: pass and move to next step, retry, or mark as failed ✅✅
7 Loop repeats All Steps 3–6 repeat for every step in the plan until all steps complete or a step fails ✅✅
8 Final result Amplify Frontend AgentCore returns { passed:true/false, steps:[...] }. UI shows pass/fail per step. Browser tab cleaned up. ✅ closes❌
🔄

Agent Loop

Claude + Strands SDK loop: think → call tool → observe result → repeat. One MCP call per plan step.

🌐

Real Browser

Actual Chromium running headed on a virtual display (Xvfb). Not a simulation — real DOM, real network requests.

📺

Live View

x11vnc → websockify → CloudFront → noVNC iframe. Every click and scroll is visible in real time.

✅

Pass / Fail

Each step is independently assessed. Claude can retry before marking a step failed. Full detail returned.
